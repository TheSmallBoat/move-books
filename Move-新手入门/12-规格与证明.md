# 进阶：规格（MSL）与 Move Prover

目标

- 介绍 Move 规格语言（MSL）与 Move Prover 的核心概念与常见用法；
- 用实例演示如何为函数与资源撰写规格（前置/后置条件、不变式、证明引理），并在代码中添加中文注释帮助读者理解。

为何要写规格

- 规格可表达函数的正确性约束（例如不会越界、不会非法运算、保持资源不变等），帮助在编译前捕获逻辑缺陷；
- Move Prover 能在数学上证明这些规格在所有可能输入下成立，从而提升合约的安全性和可审计性。

核心概念速览

- `spec` 块：在模块、函数或类型中声明规格（顶层或嵌入在函数内）；
- `requires`：前置条件（调用者必须满足）；
- `ensures`：后置条件（函数返回后必须满足）；
- `aborts_if`：声明在何种条件下函数会 abort；
- `invariant`：资源或类型的不变性声明，常用于全局资源；
- `old(expr)`：在后置条件中引用调用前的表达式值；
- `global<T>(addr)`：引用位于地址上的资源实例的规范性访问器；
- ghost（幽灵）/spec-only 定义：仅用于证明、不会编译进字节码；
- `pragma`/触发器：用于控制证明策略、避免性能陷阱或强制触发谓词。


示例 1 — 来自仓库的算术证明示例（带逐行中文注释）

下面示例摘自仓库的 `spec-proofs.md`，演示如何为一个算术函数写规格并在 `proof` 块中逐步断言中间事实：

```move
fun weighted_avg_x2(x: u64, y: u64): u64 {
    // 运行时表达式：按权重计算平均后乘以 2
    (3 * x + y) / 4 * 2
}
spec weighted_avg_x2 {
    // 规格：要求避免在中间计算中发生溢出
    requires 3 * x + y <= MAX_U64;    // 中文注：保证算术安全（防止溢出）
    // 结果等于按定义计算的值
    ensures result == (3 * x + y) / 4 * 2; // 中文注：后置条件，断言返回值
    ensures result <= 3 * x + y;            // 中文注：额外性质：结果不超过加权和
} proof {
    // proof 块用于把证明拆成若干可由 SMT 求解器验证的小步骤
    let wx = 3 * x;          // 中文注：命名中间量，便于后续推导
    let sum = wx + y;        // 中文注：总和
    let half = sum / 4;      // 中文注：取商，注意整数除法截断语义
    assert half <= sum;      // 中文注：断言：商不会大于被除数
    assert half * 2 <= sum;  // 中文注：断言：乘回去后仍小于等于总和，为推导后置条件做准备
}
```

示例说明：上例展示了如何在 `proof` 块中用 `let` 绑定中间量，并用 `assert` 将复杂代数关系分解为易证的子事实，帮助 SMT 求解器完成最终证明。


示例 2 — 资源不变式与 `aborts_if`（来自 `spec-lang.md` 的真实片段，带注释）

以下示例来源于仓库 `spec-lang.md`，展示了 `aborts_if` 与 `invariant` 的实际使用场景，并在关键行添加中文注释：

```move
module 0x42::m {
  resource struct Counter { value: u8 }

  // Struct 级别不变式：任何 Counter 实例在“非变更时刻”都满足 value < 128
  spec Counter {
    invariant value < 128; // 中文注：在构造（pack）点会检查，在变更期间不保证
  }

  // 增加计数的函数（可能会因溢出而 abort）
  spec increment {
    // 说明什么时候函数会 abort：若资源不存在
    aborts_if !exists<Counter>(a); // 中文注：若不存在则会失败
    // 若希望更精确地描述 abort 行为，需要补充可能的溢出条件
    aborts_if global<Counter>(a).value == 255; // 中文注：示例：到达上限时也会 abort
  }
}
```

说明：本例直接摘取自原始文档以保持语义准确；注释强调了 invariants 的检查时机（pack 点）以及 `aborts_if` 在多个子句时的 OR 语义。


示例 3 — 依赖 Lemma 与可信假设（摘自 `spec-proofs.md`，带注释）

下面示例展示了何时使用 `assume [trusted]`，并在注释中说明其使用风险：

```move
fun div3_le(x: u64): u64 { x / 3 }
spec div3_le {
    ensures result <= x; // 中文注：期望除以 3 后不超过原值
} proof {
    // 这里使用 Trusted assume，因为 SMT 可能不会自动推导出某些除法不等式
    assume [trusted] x / 3 <= x; // 中文注：trusted 假设是有风险的，必须谨慎使用并记录理由
}
```

说明：`assume [trusted]` 使证明器接收一个未经证明的前提，适用于已知数学事实但求解器难以证明的场景。尽量避免滥用。


示例 4 — 逐步代数推导与 `calc`（摘自 `spec-proofs.md`，带注释）

`calc` 语句便于表示一系列的等式/不等式链，每一步会生成单独的验证条件：

```move
fun square_plus_one(x: u64): u64 {
    (x + 1) * (x + 1)
}
spec square_plus_one {
    requires x + 1 <= 4294967295; // 中文注：防止中间乘法溢出（示例常数）
    ensures result == (x + 1) * (x + 1);
} proof {
    let y = x + 1;            // 中文注：定义中间量 y = x + 1
    let r = y * y;            // 中文注：r 为 y 的平方
    assert r == (x + 1) * (x + 1); // 中文注：重写以便 SMT 能链式推导
    post assert r == result;  // 中文注：在返回点断言 r 与 result 相等
}
```

说明：通过 `let`、`assert` 与 `post`，可以把复杂证明分成入口（entry）与返回点（post）两段，便于局部推理。

循环与不变式示例（来自原书）

下面两个示例展示如何为循环添加 `invariant`，以及如何把不变式用于归纳证明：

```move
module 0x42::m {
    fun simple3(n: u64) {
        let x = 0;                    // 中文注：初始化计数器 x
        loop {
            spec {
                invariant x <= n;         // 中文注：循环不变式：x 在任何迭代时刻都不超过 n
            };
            if (x < n) {
                x = x + 1
            } else {
                break
            }
        };
        spec {
            assert x == n;             // 中文注：循环终止时断言满足期望的后置性质
        }
    }
}
```

说明：Move Prover 会把不变式编译为基点断言（base case）、归纳假设（assume）和归纳步的断言，从而实现对循环的归纳验证。

向量遍历与复杂不变式示例

下例展示如何在遍历向量时使用两个不变式：边界不变式和基于谓词的通用不变式（使用 `result_of` 表达闭包的行为）：

```move
fun contains(v: &vector<u64>, pred: |&u64| bool has copy + drop): bool {
        let i = 0;                       // 中文注：当前索引
        let len = std::vector::length(v); // 中文注：向量长度
        while (i < len) {
                if (pred(std::vector::borrow(v, i))) {
                        return true;
                };
                i = i + 1;
        }
        spec {
                invariant i <= len;                        // 中文注：边界不变式，索引不会超过长度
                invariant forall j in 0..i: !result_of<pred>(v[j]); // 中文注：在已检查过的前缀上，谓词均为 false
        };
        false
}
spec contains {
        requires forall x in 0..len(v): !aborts_of<pred>(v[x]); // 中文注：要求谓词在所有索引上不会 abort
        aborts_if false;                                       // 中文注：函数不会 abort
        ensures result == (exists k in 0..len(v): result_of<pred>(v[k])); // 中文注：返回值等价于存在性谓词
}
```

说明：在该示例中，`result_of<pred>` 可以在不内联闭包实现的情况下描述闭包的行为，便于在不透明（opaque）场景下构造不变式与后置断言。

Ghost / Spec-only 辅助定义

- 你可以在规格中定义 `spec` 函数或 ghost 变量，仅供证明使用，不影响运行时；
- 例如使用 spec-only helpers 来抽象复杂性质、降低证明复杂度。

调优证明的技巧

- 先写小而清晰的规格，先证明模块的关键函数；
- 将复杂性质拆分为多个 lemma，逐个证明；
- 使用 `TRACE()` 或 `--trace` 选项查看失败的反例；
- 对于难以收敛的谓词，添加 `pragma` 或触发器（triggers）以改善匹配行为；
- 避免在单个规格中使用过多量词或复杂高阶谓词；先把复杂部分封装成可复用的 spec 函数。

验证工具与常用命令

- 运行 Move Prover：通常在命令行使用诸如 `move prover` 或项目自带脚本；
- 常用选项：`--trace`（打印反例 trace）、`--timeout`（延长超时时间）、`--files`（指定要证明的文件）；
- 逐步启用证明：先对关键模块开启，再扩大到整个代码库，减少验证负担。
