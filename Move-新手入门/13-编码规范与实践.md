# 编码规范与实践

目标：提供实用的编码约定、命名建议与安全审计提示，帮助工程化使用 Move。

命名与风格

- 模块名使用小写与下划线；`struct` 类型名大写开头。
- 常量使用大写蛇形：`const MAX_SUPPLY: u64 = 1000;`。

安全审计清单（快速）

- 检查 `acquires` 是否准确。
- 验证所有 `assert!` 与 `abort` 语句是否有合理错误码。
- 注意边界条件（索引、除法、溢出）。

性能与可升级性

- 避免把大量数据保存在单个资源，考虑分表或分区设计。
- 设计升级策略与版本兼容性接口。

实践补充（来自原书的重要建议）

- 在设计不变式与 `acquires` 时，同时考虑 `disable_invariants_in_body` 与 `delegate_invariants_to_caller` 两个 pragma 的语义，以避免在初始化序列化时产生验证噪音；
- 对于复杂循环，尽早为关键循环编写不变式并记录证明意图，便于后续使用 Move Prover 或 AI 工具生成辅助 lemma；
- 在公开 API 中尽量避免使用 `aborts_if_is_partial = true`，以降低未来代码修改引入未声明 abort 情形的风险；

团队实践

- 使用单元测试与规格证明结合的方式进行代码审查。
- 模块化设计、最小化公有面。

附录：常见反模式

- 直接暴露资源字段而不是通过访问函数。
- 在核心逻辑中使用 `abort` 而不提供错误码常量或文档。

练习

- 审计一个小模块，列出至少 5 个潜在问题与修复建议。
