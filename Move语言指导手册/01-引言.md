# 01 引言：Move 到底解决什么问题

Move 是一门以“资源”为中心设计的语言。它最重要的出发点不是“能写程序”，而是“能安全地表达资产、状态和它们的转移”。这也是它和很多通用语言最大的不同：在 Move 里，有些值不是普通数据，而是必须被认真对待的资源。

你可以先记住三句话：

- 资源不会被随便复制，也不会被随便丢弃。
- 程序的变化通常是事务性的，要么整体成功，要么整体回滚。
- 模块负责定义类型和规则，脚本或入口函数负责触发一次执行。

这套设计非常适合资产、余额、凭证、权限这类对象，因为这些对象最怕“被悄悄复制一份”或者“被不小心删除”。Move 把这类风险放进类型系统里提前处理，而不是等运行时再靠约定补救。

## 先理解资源意识

在普通语言里，你可以把一个值复制给多个变量，然后各自修改。在 Move 里，是否能复制、是否能丢弃、是否能放进结构体字段、是否能进全局存储，都由类型能力明确控制。也就是说，类型不只是“这个值长什么样”，还负责“这个值能被怎样使用”。

一个很短的例子可以帮助你建立直觉：

```move
module 0x42::coin {
    struct Coin has key { value: u64 }
}
```

这里的 `Coin` 不是普通整数，也不是一个可以随意传来传去的临时对象。它代表一个资源类型，能够被存进全局存储，前提是相关规则都满足。

## 为什么要这样设计

Move 的目标不是让语法看起来像熟悉的脚本语言，而是让语言本身成为安全边界的一部分。资源模型、能力系统、模块封装、全局存储和规格语言，实际上是在为同一个目标服务：让“谁能改什么、什么时候能改、改完后还能不能保持正确”变成可检查的规则。

对新手来说，最容易误解的地方是把 Move 当作“另一种命令式语言”。更准确的理解是：Move 仍然有变量、分支、循环和函数，但它对数据所有权、资源转移和存储访问有更严格的约束。这些约束不是负担，而是它最核心的安全语义。

## 本书怎么读

接下来的章节会先把代码形状、基础类型和控制流讲清楚，再逐步进入函数、模块、结构体、资源与存储。后面的章节会越来越接近 Move 的真正核心：你如何借助类型系统写出不会轻易犯错的代码。

## 再补一层理解

原书对 Move 的定位很明确：它不是为了“写起来像普通脚本语言”，而是为了让数字资产和状态迁移有可验证的边界。读这本手册时，建议你始终带着这个问题：这个值到底只是普通数据，还是一个必须被类型系统严肃对待的资源？

```move
module 0x42::wallet {
    struct Wallet has key {
        balance: u64,
    }
}
```

这个例子看起来非常普通，但语义并不普通。`Wallet` 不是“一个装着余额的盒子”这么简单，它还是一个可以进入全局存储的资源根。换句话说，后面如果你看到 `move_to`、`borrow_global` 或 `move_from`，你就可以马上联想到这里的对象不是局部变量，而是有全局生命周期的状态。

## 谁更适合先读这本书

原书的读者画像其实很明确：它不是面向完全零基础的新手，而是面向“已经会一门编程语言、但还没接触过 Move”的读者。换句话说，如果你知道函数、结构体、条件分支、测试和包管理是什么，这本书会更快进入正题；如果你还不熟悉这些概念，建议先把 02 到 06 章读顺，再回头看 08 到 12 章。

## 这本书和原书的关系

原书更像一本参考手册，按章节把语言事实讲完整；而这本中文手册更像一份面向新手的导学材料。我们保留原书的关键语义和代表性例子，但会把它们重新组织成更容易消化的中文叙述。实际阅读时，你可以把原书当作“事实底座”，把中文手册当作“更容易入口的解释层”。

## 一条更实用的阅读顺序

- 先看 02，知道项目和测试怎么跑。
- 再看 03 到 06，建立语法、变量、控制流和函数的直觉。
- 接着看 07 到 10，真正进入 Move 的数据模型和资源世界。
- 然后看 11 到 12，理解测试和形式化验证怎么补强语义。
- 最后看 13 和 14，把风格、术语和回查路径补齐。

中文备注：这条顺序不是唯一正确的，但它符合原书从“语言概貌”到“安全和验证”的组织逻辑。对于第一次学习 Move 的人来说，先顺着这个逻辑走一遍，会比跳着读更稳。
