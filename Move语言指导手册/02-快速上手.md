# 02 快速上手：先把包、模块和测试跑起来

如果你第一次接触 Move，最重要的不是先背语法，而是先知道一个 Move 项目通常长什么样。Move 的代码通常按“包”组织，包里放模块、测试、示例脚本和配置文件。你写的不是一个零散文件，而是一个带依赖和命名空间的项目。

## 先看结构

一个最常见的包大致长这样：

```text
Move.toml
sources/
tests/
scripts/
examples/
```

- `Move.toml` 是包清单，用来描述包名、版本、地址和依赖。
- `sources/` 放正式模块和脚本。
- `tests/` 放测试代码。
- `scripts/` 和 `examples/` 是否存在，取决于你的工具链和项目组织方式。

## 一个最小心智模型

你可以把自己的工作流理解成四步：

1. 定义包和地址。
2. 编写模块，声明类型和函数。
3. 通过测试验证行为。
4. 再把模块交给运行环境执行或发布。

如果你使用的是 Aptos CLI，通常会看到类似下面的命令：

```shellscript
> aptos move init --name HelloMove
> aptos move test
```

这些命令只是一个常见示例。更广义地说，任何 Move 工具链都需要完成同样的两件事：构建包，以及运行测试或入口逻辑。

## 先写一个最小模块

下面这个例子展示了一个最简单的模块骨架：

```move
module 0x42::counter {
    struct Counter has key { value: u64 }

    public fun new(value: u64): Counter {
        Counter { value }
    }
}
```

这段代码不涉及全局存储，只是先定义一个可复用的数据类型和一个构造函数。新手在上手时，最好先把“模块里定义类型和函数”这个结构记牢，再去学资源发布、借用和测试。

## 为什么要先跑测试

Move 的很多错误不是“语法错”，而是“语义不满足”。比如：

- 资源被错误复制。
- 变量的类型不匹配。
- 访问了不存在的全局资源。
- 断言条件没有成立。

测试能尽早把这些问题暴露出来，所以这本手册后面会一再强调：不要只看代码能不能编译，还要看它是否表达了你真正想要的语义。

## 一个更像真实项目的包清单

原书的项目组织方式是通过 `Move.toml` 把包名、地址和依赖串起来的。下面是一个非常小的例子：

```toml
[package]
name = "hello_move"
version = "0.0.0"

[addresses]
hello_move = "_"
```

这里的 `hello_move` 只是一个名字占位符，真正的地址值通常由构建环境在编译时填入。这个机制很重要，因为它让源码可以保持可移植，而不是把部署地址写死在逻辑里。

## 你跑测试时会看到什么

原书里给出的测试输出说明了一个很实用的事实：Move 测试不是“黑盒地告诉你通过了”，而是会告诉你每个用例是 `PASS`、`FAIL` 还是 `TIMEOUT`。

```text
> aptos move test
[ PASS    ] 0x1::my_module::make_sure_non_zero_coin_passes
[ PASS    ] 0x1::my_module::make_sure_zero_coin_fails
Test result: OK. Total tests: 2; passed: 2; failed: 0
```

中文上最该注意的是：测试成功并不等于逻辑完美，只是说明当前用例覆盖到的路径没有暴露问题。所以后面的断言和规格章节，实际上是在补测试无法覆盖到的那一部分语义。

## 一个更像真实项目的 `Move.toml`

原书一直强调，Move 的项目不是单个文件，而是一个带清单、地址和依赖的包。下面这个例子比最小清单更接近真实项目：

```toml
[package]
name = "hello_move"
version = "0.0.0"

[addresses]
std = "0x1"
hello_move = "_"

[dependencies]
MoveStdlib = { local = "../move-stdlib", addr_subst = { "std" = "0x1" } }
```

中文备注：`addresses` 的作用，是把源码里的命名地址和实际编译地址连接起来；`dependencies` 的作用，则是把别的包带进来。你不必一开始就把所有细节背熟，但至少要知道它们分别控制“源码看到什么地址”和“编译时依赖什么代码”。

## `aptos move test` 不只是跑一遍

原书把测试命令和覆盖率命令单独拿出来，是因为它们不是同一层次的工具。

```text
> aptos move test -f zero_coin
```

```text
> aptos move test --coverage
```

```text
> aptos move test -h
> aptos move coverage -h
```

中文备注：`-f` 适合局部筛选，`--coverage` 适合检查测试是否覆盖到关键路径，而 `-h` 则是你在忘记选项时最应该先看的东西。原书把这些命令放在一起，是为了让你形成“测试不是一个动作，而是一套工作流”的概念。

## 从包到入口的心智模型

如果你把 Move 项目放进一个更大的工作流里，可以把它理解成：先用 `Move.toml` 告诉构建系统“这个包叫什么、依赖什么、地址怎么替换”，再通过模块和脚本把行为组织起来，最后用测试和入口函数验证执行结果。这个顺序很重要，因为它决定了你是在“写一个文件”，还是在“维护一个可以发布的包”。
