# 05 控制流：条件、模式和循环

Move 的控制流看起来很熟悉，但它和很多语言不同的一点是：`if` 和 `match` 往往是表达式，不只是“跳转语句”。这让你可以把分支结果直接作为值使用。

## `if` 和 `else`

`if` 的条件必须是 `bool`。如果两边分支都返回值，那么这两个分支的类型必须兼容。

```move
script {
    fun max(a: u64, b: u64): u64 {
        if (a > b) a else b
    }
}
```

如果没有写 `else`，Move 会把缺失的分支看成 `()`。这就是为什么“有时能写成表达式，有时不能”的根本原因：两边的返回类型要能对齐。

## `match`

`match` 是多路分支工具。它会按顺序检查每个分支，先匹配到的分支生效。对新手来说，最值得先记住的就是两种常见用途：

- 按字面值分支，例如匹配整数、布尔值或字节串。
- 按范围分支，例如区分负数、零和正数。

```move
fun sign(x: i32): u8 {
    match (x) {
        ..0 => 0,
        0 => 1,
        1.. => 2,
    }
}
```

`match` 还可以和引用一起使用；这让你可以在不复制值的前提下做模式判断。结构体和枚举的模式会在后面的章节中进一步展开。

## `while` 和 `loop`

Move 支持条件循环和无限循环：

```move
script {
    fun sum_to(n: u64): u64 {
        let sum = 0;
        let i = 0;
        while (i <= n) {
            sum = sum + i;
            i = i + 1;
        };
        sum
    }
}
```

`loop` 适合“直到满足某个条件才退出”的场景，通常会和 `break` 一起使用。

## `break`、`continue` 和循环标签

如果你需要在嵌套循环中跳出外层循环，可以给循环加标签。Move 2.1 以后支持这种写法：

```move
script {
    fun example(x: u64) {
        'outer: while (x > 10) {
            if (x % 2 == 0) {
                break 'outer;
            } else {
                continue 'outer;
            }
        }
    }
}
```

这类语法的意义不在于“更炫”，而在于它能让控制流意图更明确，尤其是在处理查找、扫描和重试逻辑时。

## 读 `match` 的一个小提醒

`match` 的顺序很重要。你不仅要想清楚“有哪些分支”，还要想清楚“先写哪个分支”。在复杂模式里，先写更具体的匹配，再写兜底分支，通常更安全也更容易读懂。

## 再看两个更原书化的 `match` 例子

原书把 `match` 讲得很细，其中最值得借鉴的是“按值分类”和“通过引用分类”这两类写法。

```move
fun classify(x: i32): u8 {
    match (x) {
        ..0 => 0,   // 负数
        0 => 1,     // 零
        1.. => 2,   // 正数
    }
}

fun bool_ref(b: &bool): u64 {
    match (b) {
        true => 1,
        false => 0,
    }
}
```

中文备注：第一段的关键不是“写法酷”，而是让你把数值边界一次说清楚；第二段的关键不是“引用也能匹配”，而是提醒你在不复制原值的情况下也能分支判断。对于资源类型来说，这种写法尤其有价值，因为你不会为了判断一个状态而意外触发复制。

## 循环标签再强调一次

如果你在嵌套循环里做查找或重试，循环标签通常会比在内层循环里手写复杂标志更清楚。

```move
script {
    fun scan(mut n: u64) {
        'outer: loop {
            if (n == 0) {
                break 'outer;
            };
            n = n - 1;
        }
    }
}
```

中文备注：标签不是为了“多一种语法”，而是为了让“我想退出哪一层循环”这件事写得一眼可见。

## `abort` 是控制流的一部分

原书把 `abort` 放在控制流章节附近，是因为它和 `return` 一样，都是让执行提前结束的表达式；不同的是，`abort` 会回滚整个事务。

```move
script {
    use std::vector;

    fun pop_twice<T>(v: &mut vector<T>): (T, T) {
        if (vector::length(v) < 2) abort 42;

        (vector::pop_back(v), vector::pop_back(v))
    }

    fun check_vec(v: &vector<u64>, bound: u64) {
        let i = 0;
        let n = vector::length(v);
        while (i < n) {
            let cur = *vector::borrow(v, i);
            if (cur > bound) abort 42;
            i = i + 1;
        }
    }
}
```

中文备注：这一段特别重要，因为它让你看到 Move 的“失败”不是局部失败，而是事务失败。只要 `abort` 发生，之前对全局存储做过的修改都会被撤销。

## `assert!` 其实就是更常用的 `abort`

```move
script {
    use std::vector;

    fun pop_twice<T>(v: &mut vector<T>): (T, T) {
        assert!(vector::length(v) >= 2, 42);
        (vector::pop_back(v), vector::pop_back(v))
    }
}
```

中文备注：原书之所以强调 `assert!`，是因为它把“先检查条件，再决定是否中止”这件事写得更清楚，也更常见。对于需要参数化错误码或错误消息的场景，这种写法尤其自然。

## `match` 的守卫条件

`match` 不只是“按模式拆分”，还可以在分支后面加守卫条件 `if guard`。这让你可以在模式匹配基础上再加一层条件判断。

```move
fun status_message(code: u16): vector<u8> {
    match (code) {
        200 => b"OK",
        404 => b"Not Found",
        500 => b"Server Error",
        _ if code < 100 => b"Informational",
        _ => b"Unknown",
    }
}
```

中文备注：原书把这个写法和普通 `if/else` 对比，是为了说明 `match` 并不只是“更长的 if”，它更适合把多种情况按结构分类，再按条件微调。
