# 06 函数与模块：代码如何组织与暴露

函数和模块是 Move 里最重要的组织单元。你可以把模块理解成“类型和规则的边界”，把函数理解成“在这个边界里公开或内部可调用的行为”。

## 函数长什么样

Move 函数通常写成这样：

```move
fun name<T>(x: T, y: u64): T {
    x
}
```

这里最值得先记住的是三件事：

- 函数可以有类型参数，也可以没有。
- 函数的返回值可以是单个值，也可以是元组形式的多个值。
- 如果函数访问了全局存储里的资源，通常需要写 `acquires`。

## 可见性

模块函数默认是模块私有的，也就是只有同一模块内部能调用。想让别的模块或脚本调用，就要显式声明可见性。

- `public`：任何模块或脚本都能调用。
- `package`：同一个包内可见，编译期会被降为 friend 语义。
- `friend` / `public(friend)`：只允许友元模块调用，并且必须在同地址范围内。

对新手来说，可以先把它们理解为三层门禁：公开、包内、友元。

## `entry`

`entry` 用来标记“可以作为入口直接调用”的函数。它更像模块里的主入口，而不是普通内部工具函数。

- `entry` 函数可以是公开的，也可以是内部的。
- `entry` 函数不能返回值。
- `entry` 函数能接受的参数类型是受限的，通常是基础类型、`signer` 引用、合规的向量和少数标准库类型。

你可以把它理解成“受控的事务入口”。

## `self` 受体写法

如果函数的第一个参数命名为 `self`，就可以用点语法调用：

```move
module 0x42::example {
    struct S {}

    fun foo(self: &S, x: u64) {
        // ...
    }

    fun demo() {
        let s = S {};
        s.foo(1);
    }
}
```

`s.foo(1)` 实际上等价于 `foo(&s, 1)`。编译器会自动补上引用运算符，所以这种写法更像面向对象语言里的方法调用，但底层仍然是普通函数。

## 模块和脚本

模块是可复用的库代码，脚本是一次性执行片段。模块负责定义类型、函数和常量；脚本更像“把某个公开接口调用一次”。

一个典型模块会这样组织：

```move
module 0x42::bank {
    use std::signer;

    struct Balance has key { value: u64 }

    public fun read(addr: address): u64 acquires Balance {
        borrow_global<Balance>(addr).value
    }
}
```

模块名通常以小写字母开头，文件名也通常和模块名保持一致。脚本则更接近“调用一个主函数”，通常不会定义资源类型。

## `acquires`

如果函数访问了全局资源，就应该把它依赖的资源类型写进 `acquires`。这样做不是为了装饰，而是为了让编译器和读者都知道：这个函数会接触全局存储。

```move
module 0x42::bank {
    struct Balance has key { value: u64 }

    public fun get(addr: address): u64 acquires Balance {
        borrow_global<Balance>(addr).value
    }
}
```

在较新的工具链里，`acquires` 可以由编译器推断，但把它写出来通常更清晰，也更适合审计和形式化验证。

## 函数值：进阶但重要

Move 还支持函数值，也就是“把函数当成值来传递”。这通常用于动态分发、回调或者更通用的组合式设计。函数值可以来自函数名，也可以来自 lambda。

如果你以后要把函数值存进结构体，记住一条规则：想让它可存储，底层函数通常需要是 persistent；公开函数默认满足这一点，非公开函数则需要显式标记。

这个特性对新手来说不必第一天就掌握，但你至少要知道：Move 不只是“调用函数”，它也可以“携带函数”。

## 原书里最值得保留的几个细节

### 1. `entry` 不是“只能入口调用”的硬限制

原书强调得很清楚：`entry` 函数可以作为模块入口，但它本身仍然可以被其他 Move 函数调用。也就是说，`entry` 更像“允许从交易开始的位置”，而不是“禁止内部复用”的标记。

```move
module 0x42::example {
    public entry fun public_entry() {}

    entry fun internal_entry() {}

    fun call_both() {
        public_entry();
        internal_entry();
    }
}
```

中文备注：新手常常把 `entry` 理解成“只能从外部执行一次”，但原书并不是这个意思。它的核心作用是告诉运行环境：这个函数可以作为事务入口被调度。

### 2. `package` 和 `friend` 是两种不同的门禁

```move
module 0x42::m {
    friend 0x42::n;

    package fun package_only(): u64 { 1 }
    public(friend) fun friend_only(): u64 { 2 }
}

module 0x42::n {
    fun read(): u64 {
        0x42::m::package_only() + 0x42::m::friend_only()
    }
}
```

中文备注：`package` 面向“同一个包”，`friend` 面向“显式授权的模块”。它们看起来都像“限制访问”，但边界并不一样。原书特别强调，`package` 在编译期会被降成 friend 语义，这意味着它本质上仍然是受 VM 认可的友元访问。

### 3. 函数值和 `#[persistent]`

```move
#[persistent]
fun is_even(x: u64): bool {
    x % 2 == 0
}

fun is_odd(x: u64): bool {
    x % 2 == 1
}

fun demo() {
    let f: |u64|bool has copy+drop+store = is_even;
    let g: |u64|bool has copy+drop = is_odd;
    assert!(f(2), 0);
}
```

中文备注：原书用函数值强调了一件事——Move 也能做动态分发。`#[persistent]` 的意义在于保证未来还能安全恢复这个函数值，所以它比“只想让某个函数可存储”更接近一种长期可用性承诺。

### 4. `entry` 参数类型有边界

原书说明 `entry` 函数能接受的参数并不是任意类型。对新手来说，最实用的记忆方式是：能从交易参数安全构造出来的值才行，资源类型不能直接从参数里凭空造出来。

```move
public entry fun call_me(amount: u64, flag: bool) {
    let _ = amount + if (flag) 1 else 0;
}
```

中文备注：这里的重点不是“能不能写入口函数”，而是“入口函数能接收的内容是否足够受控”。这正是 Move 防止资源被随意伪造的关键之一。

## 受体调用和泛型方法风格

原书把 `self` 受体写法和泛型函数放在一起讲，是因为点调用不是一种新的语义，只是更短的调用写法。

```move
module 0x42::example {
    struct S {}

    fun foo(self: &S, x: u64) {
        // ...
    }

    fun demo() {
        let s = S {};
        s.foo(1);
    }
}
```

```move
module 0x42::example {
    fun bar() {
        let v = vector[1, 2, 3];
        let e1 = v.remove(0);
        let e2 = v.remove::<u8>(0);
    }
}
```

中文备注：原书在这里想说明两件事。第一，点语法只是糖，背后还是普通函数调用；第二，泛型实参既可以推断，也可以显式写出。对读者来说，关键是理解“调用方式变了，但类型系统规则没变”。

## 函数值、lambda 和闭包

原书的函数值章节是这一章里最容易被忽视、但也很有代表性的部分。它说明 Move 不只是支持“调用函数”，也支持“把函数作为数据传递”。

```move
public fun is_even(x: u64): bool { x % 2 == 0 }

fun demo() {
    let f: |u64|bool has copy+drop = is_even;
    assert!(f(1) == f(2), 0);
}
```

```move
struct Predicate<T>(|&T|bool) has copy;

fun demo2() {
    let f: Predicate<u64> = |x| *x > 0;
    assert!(f(&22), 0);
}
```

```move
#[persistent]
fun is_odd(x: u64): bool { x % 2 == 1 }

fun demo3() {
    let g: |u64|bool has copy+drop+store = is_odd;
}
```

中文备注：这里的函数值不只是“能放进变量里”，更重要的是它们有清晰的能力边界。由函数名构造的函数值会继承底层函数的可复制、可丢弃或可存储属性；由 lambda 构造的函数值则像一个轻量闭包，参数和捕获范围都受到语言规则约束。`Predicate<T>` 说明了函数值还可以被结构体封装成更有名字的类型，而 `#[persistent]` 则把“可存储”从公开可见性里拆出来，避免把 API 面扩大到不必要的程度。

## `#[persistent]` 的真正用途

原书把 `#[persistent]` 和 `public`、`entry` 区分开，是因为“可存储”不等于“公开 API”。如果你只想让一个函数值能被安全保存，`#[persistent]` 往往比把函数放成 `public` 或 `entry` 更合适。

中文备注：这条规则对设计回调、策略对象或可恢复函数值特别有用。它让你在不扩大外部可调用面的前提下，仍然能把函数当成长期存储的数据。

## 再补一个 `acquires` 的调用链例子

```move
module 0x42::example {
    struct Counter has key { value: u64 }

    public fun increment(addr: address) acquires Counter {
        let value_ref = &mut borrow_global_mut<Counter>(addr).value;
        *value_ref = *value_ref + 1;
    }

    fun call_increment(addr: address): u64 acquires Counter {
        increment(addr);
        borrow_global<Counter>(addr).value
    }
}
```

中文备注：这类例子和 08 章里的资源访问是互相呼应的。原书想让你看到的是：函数的可见性、`acquires` 和资源定义模块是一起工作的，不是各讲各的三套规则。
