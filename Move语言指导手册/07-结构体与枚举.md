# 07 结构体与枚举：自定义数据类型

如果说原始类型是 Move 的“字母表”，那么结构体和枚举就是它的“词汇表”。你会用它们来表达余额、订单、配置、状态机和各种领域对象。

## 结构体是什么

结构体是用户定义的数据类型，字段可以是非引用类型。结构体必须定义在模块内部。

```move
module 0x2::m {
    struct Foo { x: u64, y: bool }
}
```

默认情况下，结构体是“线性且短暂的”：

- 不能复制，除非显式获得 `copy` 能力。
- 不能丢弃，除非显式获得 `drop` 能力。
- 不能作为全局存储的根，除非有 `key` 能力。

这就是为什么资源类型通常会用结构体来表达：你必须显式处理它们，而不能随手忽略。

## 能力决定行为

结构体可以通过 `has copy, drop, store, key` 等能力放宽限制。能力不是装饰性注解，而是类型安全规则的一部分。

你可以先记住最常见的一条经验：

- 如果一个结构体代表普通数据，就可能需要 `copy` 和 `drop`。
- 如果一个结构体代表资源，就通常不要给它 `copy` 和 `drop`。

## 创建和拆解结构体

创建结构体值叫 pack，拆解结构体值叫 unpack。

```move
module 0x2::m {
    struct Foo has copy, drop { x: u64, y: bool }

    fun demo() {
        let foo = Foo { x: 3, y: false };
        let Foo { x, y: foo_y } = foo;
    }
}
```

如果字段名和局部变量名相同，可以写成字段名简写：`Foo { x }` 等价于 `Foo { x: x }`。这个技巧很常见，也很适合让代码更短更清楚。

还可以用 `_` 丢掉不关心的字段，或者用嵌套模式拆解更复杂的数据。

## 借用字段

结构体字段可以被借用：

```move
module 0x2::m {
    struct Foo { x: u64, y: bool }

    fun demo() {
        let foo = Foo { x: 3, y: true };
        let x_ref = &foo.x;
        let y_ref = &mut foo.y;
    }
}
```

对新手来说，这里最重要的理解是：引用并不会复制整个结构体，只是把访问权借出去。引用到引用、把引用塞进结构体字段、或者让全局存储里出现引用，都是 Move 明确禁止或强烈限制的。

## 位置结构体

除了命名字段结构体，Move 还支持位置结构体，也就是字段按顺序编号的写法：

```move
module 0x2::m {
    struct Pair(u64, bool);

    fun demo() {
        let value = Pair(1, true);
        let Pair(number, flag) = value;
    }
}
```

位置结构体适合字段很少、表达很紧凑的类型。它们的字段也可以通过 `.0`、`.1` 这样的方式访问。

## 枚举是什么

枚举允许一个类型有多个变体。每个变体可以有自己的字段布局：

```move
enum Shape {
    Circle { radius: u64 },
    Rectangle { width: u64, height: u64 },
}
```

枚举特别适合表达“当前处于哪一种状态”。例如一个版本化数据类型可以有 `V1`、`V2` 等多个变体。

Move 2.0 以后，枚举也支持位置变体：

```move
enum Result<T> has copy, drop, store {
    Err(u64),
    Ok(T),
}
```

枚举值的构造通常要写成 `TypeName::Variant`。如果变体没有字段，可以直接写成 `Color::Blue` 这样的形式。

## 模式匹配

结构体和枚举都可以在 `match` 中被匹配。对于新手来说，模式匹配的价值在于：你可以在编译期就把“各种可能性”写出来，而不是靠一堆手写条件漏掉分支。

```move
fun describe(c: Color): u8 {
    match (c) {
        Color::Red => 1,
        Color::Blue => 2,
        Color::Green => 3,
    }
}
```

## 可见性修饰

在较新的 Move 版本里，结构体和枚举也可以带 `public`、`package`、`friend` 等可见性修饰。这样一来，外部模块可以在受控范围内构造、拆解和访问字段。

这类能力很强，但也意味着你要更谨慎地设计 API：越多操作暴露给外部，越要确认你的类型不变量仍然能被维持。

## 更多原书式例子

### 1. 嵌套拆解很常见

```move
module 0x2::m {
    struct Foo { x: u64, y: bool }
    struct Bar { foo: Foo }

    fun demo() {
        let bar = Bar { foo: Foo { x: 3, y: false } };
        let Bar { foo: Foo { x, y } } = bar;
    }
}
```

中文备注：嵌套模式并不是“炫技写法”，而是让你一次性把深层结构拆开。原书在这里想传递的重点是：结构体既能装数据，也能在模式匹配里被精确拆分。

### 2. 位置结构体很适合轻量封装

```move
module 0x2::m {
    struct Pair(u64, u8);

    fun demo() {
        let value = Pair(1, 2);
        let Pair(number, flag) = value;
        let first = value.0;
        let second = value.1;
    }
}
```

中文备注：位置结构体最适合字段很少、语义很紧凑的场景。原书特别提示过，`value.0`、`value.1` 这样的字段访问可以让你在不解构整个值的情况下直接读到对应部分。

### 3. 枚举是天然的状态机表达方式

```move
enum Result<T> has copy, drop, store {
    Err(u64),
    Ok(T),
}

fun use_result(r: Result<u64>): u64 {
    match (r) {
        Result::Err(code) => code,
        Result::Ok(v) => v,
    }
}
```

中文备注：枚举的价值在于“一个类型表示多个状态”。原书用 `VersionedData` 和 `Result<T>` 这类例子，正是想说明：只要状态空间有限且明确，枚举通常比用一堆布尔字段更安全。

### 4. 部分模式和 `..` 很实用

```move
module 0x2::m {
    struct Point { x: u64, y: u64, z: u64 }

    fun demo(p: Point) {
        let Point { x, .. } = p;
    }
}
```

中文备注：`..` 的作用是忽略你暂时不关心的字段。原书把它和“只拆出需要的部分”放在一起讲，是因为这能让代码同时保持简洁和语义完整。

## 字段借用与局部更新

原书在结构体章节里反复展示的一件事，是“借用字段”和“修改字段”可以非常局部地进行，而不必把整个结构体复制出来。

```move
module 0x2::m {
    struct Foo { x: u64, y: bool }
    struct Bar { foo: Foo }

    fun example() {
        let foo = Foo { x: 3, y: true };
        let foo_ref: &Foo = &foo;
        let y: bool = foo_ref.y;
        let x_ref: &u64 = &foo.x;

        let x_ref_mut: &mut u64 = &mut foo.x;
        *x_ref_mut = 42;
    }

    fun nested() {
        let foo = Foo { x: 3, y: true };
        let bar = Bar { foo };
        let x_ref = &bar.foo.x;
        let foo_ref = &bar.foo;
        let x_again = foo_ref.x;
    }
}
```

中文备注：`foo_ref.y`、`&foo.x` 和 `&mut foo.x` 看起来只差几个符号，但语义差别很大。原书在这里想让你明确：Move 的引用不是“复制一个备份”，而是“把访问权借出去”。当字段本身有 `copy` 能力时，读取会复制字段值；但这不代表你可以绕开资源语义。

## 结构体的解构方式再补几个细节

原书展示了字段名简写、通配符和嵌套模式，这些写法在 Move 中非常常见：

```move
module 0x2::m {
    struct Foo has copy, drop { x: u64, y: bool }
    struct Bar { foo: Foo }

    fun example_destroy_foo() {
        let foo = Foo { x: 3, y: false };
        let Foo { x, y: foo_y } = foo;
    }

    fun example_destroy_foo_wildcard() {
        let foo = Foo { x: 3, y: false };
        let Foo { x, y: _ } = foo;
    }

    fun example_destroy_bar() {
        let bar = Bar { foo: Foo { x: 3, y: false } };
        let Bar { foo: Foo { x, y } } = bar;
    }
}
```

中文备注：原书之所以不断使用这些例子，是因为它希望你习惯一种思考方式：不是先“把值整体拿出来”，而是先决定“我到底需要哪些字段”。这和资源安全是同一件事的两面。

## 位置结构体和部分模式

位置结构体是原书里另一个很重要的扩展点。它既能让类型定义更紧凑，也能让局部模式更自然。

```move
module 0x2::m {
    struct Pair(u64, u8);
    struct TypeTag has copy, drop;

    fun work() {
        let value = Pair(1, true);
        let Pair(number, boolean) = value;
        assert!(number == 1 && boolean == true);

        let Pair(first, ..) = value;
        let Pair(.., last) = value;
    }
}
```

中文备注：`..` 不只是“省略号”，它表达的是“我有意识地不处理这些字段”。原书特别强调，部分模式目前不能用在赋值左侧，这说明它仍然是一个受约束的匹配工具，而不是所有地方都能随便使用的通配写法。

## 枚举的构造、选择和匹配

原书对枚举写得非常细，因为它们不仅能表达“多种状态”，还能支持字段选择、`is` 测试和部分模式。

```move
enum Shape {
    Circle { radius: u64 },
    Rectangle { width: u64, height: u64 },
}

enum Color has copy, drop, store, key { Red, Blue, Green }

enum VersionedData has key {
    V1 { name: String },
    V2 { name: String, age: u64 },
}

fun demo(s: String, data1: VersionedData, data2: VersionedData) {
    let c = Color::Blue;
    let v1 = VersionedData::V1 { name: s };
    let v2 = VersionedData::V2 { name: s, age: 20 };

    assert!(data1.name == data2.name, 0);
    assert!(v2.age == 20, 0);

    if (data1 is VersionedData::V1) {
        // 这里可以安全地把 data1 视为 V1
    }

    match (data2) {
        V1 { .. } => 1,
        V2 { name, age: 0..=17 } => 2,
        V2 { name, age } => 3,
    };
}
```

中文备注：原书特别提醒过，枚举变体名通常需要用 `TypeName::Variant` 这种限定形式；在某些 `match` 场景里，编译器可以根据上下文推断出枚举类型，所以 `V1{..}` 这种简写也可能成立。但变体别名目前还不能通过 `use` 来做，这一点很容易被忽略。

## 枚举的 let 解构和销毁

枚举不仅能在 `match` 中处理，也可以在 `let` 里解构。但原书提醒得很清楚：`let` 解构如果写错变体，运行时会 abort，所以更适合你已经确定分支的场景。

```move
enum Shape {
    Circle { radius: u64 },
    Rectangle { width: u64, height: u64 },
}

fun destroy_empty(self: Shape) {
    match (self) {
        Shape::Circle { radius } => assert!(radius == 0, 0),
        Shape::Rectangle { width, height: _ } => assert!(width == 0, 0),
    }
}
```

中文备注：原书用这类例子说明，`match` 是“穷尽式处理”，`let` 是“我已经知道这就是这个变体”。前者更安全，后者更短，但要求你对输入更有把握。

## 可见性、事务参数和升级

Move 2.4 开始，结构体和枚举可以有 `public`、`package`、`friend` 等可见性修饰，但原书也明确指出了几个限制：带 `key` 能力的类型不能公开；全局存储操作仍然只在定义模块内允许；而 `public` 结构体和枚举只有在满足能力条件时，才可作为交易参数。

```move
module 0x42::types {
    public struct Point has copy, drop {
        x: u64,
        y: u64,
    }

    public enum Direction has copy, drop {
        North, South, East, West,
    }

    entry fun move_to_point(s: &signer, destination: Point) { }
    entry fun move_player(s: &signer, dir: Direction) { }
}
```

中文备注：原书把“可见性”和“可升级性”放在一起讲，是因为它们都影响模块外部怎么依赖这些类型。升级时，`public` 不能被降级，`package` 和 `friend` 之间的转换也有明确边界；对枚举来说，新增变体必须加在末尾，不能打乱已有顺序。
