# 08 资源模型与全局存储：Move 的核心约束

如果前几章是在教你“怎么写 Move”，这一章就是在教你“Move 为什么安全”。资源模型和全局存储，是 Move 语言设计里最关键的部分。

## 全局存储是什么

Move 程序的目标，是读写一个树状的持久化全局存储。你可以把它想成：每个地址下面都挂着一棵小树，树上既能存资源数据，也能存模块代码。

对普通程序来说，这可能听起来像数据库；但对 Move 来说，它是语言语义的一部分，不是外部依赖。

## 五个基本操作

Move 通过五个操作访问全局存储中的资源：

- `move_to<T>(&signer, T)`：把资源发布到签名者地址。
- `move_from<T>(address): T`：从地址移除资源并返回它。
- `borrow_global<T>(address): &T`：借用不可变引用。
- `borrow_global_mut<T>(address): &mut T`：借用可变引用。
- `exists<T>(address): bool`：检查资源是否存在。

这五个操作都和类型 `T` 绑定，而且 `T` 必须具有 `key` 能力。

## 资源不是普通值

资源之所以特殊，是因为它们代表“必须被认真追踪的状态”。因此，Move 不允许你在不经意间把资源复制一份，或者在没有处理后果的情况下把它丢掉。

这也是为什么资源通常放在自己的模块里，并通过模块暴露受控 API。外部模块不应该直接绕过定义模块的规则去操作资源。

## `acquires`

如果函数直接或间接访问了全局资源，通常要写 `acquires`。它告诉编译器和读者：这个函数会接触某些模块定义的全局状态。

```move
module 0x42::counter {
    struct Counter has key { value: u64 }

    public fun read(addr: address): u64 acquires Counter {
        borrow_global<Counter>(addr).value
    }
}
```

把 `acquires` 写出来并不只是形式要求。它还帮助你提前发现“这个函数可能和哪类全局资源发生关系”，这对于审计和证明都很重要。

## 全局引用不能逃逸

Move 允许你借用全局资源，但不允许你把指向全局存储的引用直接返回出去。原因很简单：一旦资源被后续事务移动、删除或更新，这类引用就会悬空。

所以，下面这种想法在 Move 里是被禁止的：

```move
// 不要试图把 borrow_global 得到的引用直接返回出去
fun bad(addr: address): &u64 {
    borrow_global<Something>(addr).value
}
```

Move 通过类型系统阻止这种错误，而不是等运行时再补救。

## 存储多态

Move 还允许对泛型资源做全局存储操作。也就是说，你可以把类型参数和存储访问组合起来，让同一套代码处理不同类型的资源。

这种能力通常被称为存储多态。它让 Move 的资源模型既严格，又足够灵活。

## `Counter` 示例

下面是一个简化但完整的例子：

```move
module 0x42::counter {
    use std::signer;

    struct Counter has key { value: u64 }

    public fun publish(account: &signer, value: u64) {
        move_to(account, Counter { value })
    }

    public fun get(addr: address): u64 acquires Counter {
        borrow_global<Counter>(addr).value
    }

    public fun inc(addr: address) acquires Counter {
        let value_ref = &mut borrow_global_mut<Counter>(addr).value;
        *value_ref = *value_ref + 1;
    }

    public fun remove(account: &signer): u64 acquires Counter {
        let Counter { value } = move_from<Counter>(signer::address_of(account));
        value
    }

    public fun exists_at(addr: address): bool {
        exists<Counter>(addr)
    }
}
```

这个例子值得你反复看，因为它几乎把全局存储的五种操作都串起来了。你以后写任何资源模块，都可以先问自己：我是在发布、读取、修改、删除，还是检查存在性？

## 原书里还强调了两个关键点

### 1. 泛型资源也可以进入全局存储

```move
module 0x42::example {
    struct Container<T> has key {
        t: T,
    }

    fun publish_generic_container<T>(account: &signer, t: T) {
        move_to(account, Container { t })
    }

    fun publish_instantiated_generic_container(account: &signer, t: u64) {
        move_to(account, Container<u64> { t })
    }
}
```

中文备注：这就是原书里说的“storage polymorphism”。它的意义不是语法花哨，而是让你能按类型参数组织不同的链上状态，而不用为每种状态都重复写一套相同逻辑。

### 2. 不能把全局引用带出函数

```move
module 0x42::dangling {
    struct T has key {
        f: u64,
    }

    fun borrow_then_remove_bad(a: address) acquires T {
        let t_ref: &mut T = borrow_global_mut<T>(a);
        let t = remove_t(a);
        let uh_oh = *&t_ref.f;
    }

    fun remove_t(a: address): T acquires T {
        move_from<T>(a)
    }
}
```

中文备注：原书在这里的重点非常明确——Move 不是靠运行时“希望引用还有效”，而是直接让这种悬空引用在类型层面不可通过。`acquires` 的存在，就是为了让这种危险在编译阶段被看见。

## 下标语法让存储访问更紧凑

原书后来补充了一个很实用的语法糖：你可以把全局存储当成带下标的对象来读写。

| 下标写法 | 等价操作 |
| --- | --- |
| `&T[address]` | `borrow_global<T>(address)` |
| `&mut T[address]` | `borrow_global_mut<T>(address)` |
| `T[address]` | `*borrow_global<T>(address)` |
| `T[address] = x` | `*borrow_global_mut<T>(address) = x` |
| `&T[address].field` | `&borrow_global<T>(address).field` |
| `&mut T[address].field` | `&mut borrow_global_mut<T>(address).field` |
| `T[address].field` | `borrow_global<T>(address).field` |
| `T[address].field = x` | `borrow_global_mut<T>(address).field = x` |

```move
struct R has key, drop { value: bool }

fun f1() acquires R {
    let x = &mut R[@0x1];
    x.value = false;
    assert!(R[@0x1].value == false, 0);
    R[@0x1].value = true;
    assert!(R[@0x1].value == true, 0);
}
```

中文备注：这个语法糖的价值在于，它让局部字段访问和全局资源访问看起来更像一套统一语言。读代码时，你不需要一直切换脑内模型：`T[address]` 只是更短的 `borrow_global` 访问方式。

## 泛型资源和 storage polymorphism

原书非常强调，Move 可以把类型参数和全局存储结合起来，这就是 storage polymorphism。

```move
module 0x42::example {
    struct Container<T> has key { t: T }

    fun publish_generic_container<T>(account: &signer, t: T) {
        move_to<Container<T>>(account, Container { t })
    }

    fun publish_instantiated_generic_container(account: &signer, t: u64) {
        move_to<Container<u64>>(account, Container { t })
    }
}
```

中文备注：这里的重点不是“泛型写法很灵活”，而是“全局存储可以根据类型参数选择不同资源布局”。这也是为什么原书把这一段单独点出来：它是 Move 资源建模能力的重要来源。

## `acquires` 的传递、泛型和冗余限制

```move
module 0x42::two_resources {
    struct R1 has key { f: u64 }
    struct R2 has key { g: u64 }

    fun double_acquires(a: address): u64 acquires R1, R2 {
        borrow_global<R1>(a).f + borrow_global<R2>(a).g
    }
}
```

```move
module 0x42::m {
    struct R<T> has key { t: T }

    fun acquire_generic_resource<T: store>(a: address) acquires R {
        let _ = borrow_global<R<T>>(a);
    }

    fun acquire_instantiated_generic_resource(a: address) acquires R {
        let _ = borrow_global<R<u64>>(a);
    }

    fun redundant_acquires_bad() acquires R {}
}
```

```move
module 0x42::example {
    struct Counter has key { value: u64 }

    public fun increment(addr: address) acquires Counter {
        let value_ref = &mut borrow_global_mut<Counter>(addr).value;
        *value_ref = *value_ref + 1;
    }

    fun call_increment(addr: address): u64 acquires Counter {
        increment(addr);
        borrow_global<Counter>(addr).value
    }
}
```

中文备注：原书在这里想让你掌握三件事。第一，多个资源就要写多个 `acquires`；第二，泛型资源的 `acquires` 不带具体类型实参；第三，如果函数根本没触碰资源，冗余 `acquires` 会被编译器拒绝。这个设计让依赖分析更可靠，也让审计更简单。

## 资源引用不能逃逸的两个反例

```move
module 0x42::dangling {
    struct T has key { f: u64 }

    fun borrow_then_remove_bad(a: address) acquires T {
        let t_ref: &mut T = borrow_global_mut<T>(a);
        let t = remove_t(a);
        let uh_oh = *&t_ref.f;
    }

    fun remove_t(a: address): T acquires T {
        move_from<T>(a)
    }
}
```

```move
address 0x42 {
    module m1 {
        struct T has key {}

        public fun ret_t_ref(a: address): &T acquires T {
            borrow_global<T>(a)
        }

        public fun remove_t(a: address) acquires T {
            let T {} = move_from<T>(a);
        }
    }

    module m2 {
        fun borrow_then_remove_bad(a: address) {
            let t_ref = m1::ret_t_ref(a);
            let t = m1::remove_t(a);
        }
    }
}
```

中文备注：原书用这两个反例说明，Move 对全局引用的安全要求不是“尽量避免”，而是“直接不允许返回”。这就是为什么它能在静态层面保证没有 dangling reference，也没有 null/nil 式的悬空访问。

## 再把 Counter 例子拆开看

原书写 `Counter` 不是为了展示一个简单计数器，而是为了把全局存储的五种操作一一对应起来：发布、读取、修改、删除和存在性检查。

```move
module 0x42::counter {
    use std::signer;

    struct Counter has key { i: u64 }

    public fun publish(account: &signer, i: u64) {
        move_to(account, Counter { i })
    }

    public fun get_count(addr: address): u64 acquires Counter {
        borrow_global<Counter>(addr).i
    }

    public fun increment(addr: address) acquires Counter {
        let c_ref = &mut borrow_global_mut<Counter>(addr).i;
        *c_ref = *c_ref + 1
    }

    public fun reset(account: &signer) acquires Counter {
        let c_ref = &mut borrow_global_mut<Counter>(signer::address_of(account)).i;
        *c_ref = 0
    }

    public fun delete(account: &signer): u64 acquires Counter {
        let c = move_from<Counter>(signer::address_of(account));
        let Counter { i } = c;
        i
    }

    public fun exists_at(addr: address): bool {
        exists<Counter>(addr)
    }
}
```

中文备注：如果你把这个例子读透了，后面几乎所有资源模块都能类比着理解。它的真正作用不是“写了一个可运行的模块”，而是把 Move 的资源生命周期完整地走了一遍。
