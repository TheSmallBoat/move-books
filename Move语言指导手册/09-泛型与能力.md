# 09 泛型与能力：让类型系统帮你守住边界

泛型和能力，是 Move 把“表达能力”和“安全约束”合在一起的地方。泛型让代码可以适配不同类型，能力则告诉编译器：这个类型到底允许被怎样使用。

## 泛型是什么

泛型让函数和结构体能够对不同类型工作。

```move
module 0x42::example {
    fun id<T>(x: T): T {
        x
    }

    struct Box<T> has copy, drop {
        value: T,
    }
}
```

泛型最常见的用途是写可复用库：例如向量、包装器、状态容器，以及那些希望“同一套逻辑服务多种类型”的代码。

## 类型参数和类型实参

调用泛型函数时，可以显式写类型实参，也可以交给编译器推断：

```move
let x = id<u64>(10);
let y = id(true); // 编译器推断出 T = bool
```

如果编译器无法从上下文推断类型，你就需要手动补全。最典型的例子是返回值里才出现类型参数的函数。

## 结构体里的泛型

泛型结构体可以把类型参数带进字段：

```move
module 0x2::m {
    struct Pair<T1, T2> has copy, drop {
        first: T1,
        second: T2,
    }
}
```

这让你可以写出像 `vector<T>`、`Option<T>`、`Result<T>` 这样的通用类型。

## 能力有四种

Move 的四种核心能力是：

- `copy`：值可以被复制。
- `drop`：值可以被丢弃。
- `store`：值可以被放进别的结构体字段里。
- `key`：值可以作为全局存储的根资源。

能力不是手动“贴标签”就行，而是会由字段递归决定。一个类型只有在其结构满足规则时，才配拥有某个能力。

## 为什么能力很重要

能力的意义，是把常见错误前移到编译期。

- 没有 `copy` 的值，就不能被悄悄复制。
- 没有 `drop` 的值，就不能被悄悄扔掉。
- 没有 `store` 的值，就不能被塞进普通结构体里长期保存。
- 没有 `key` 的值，就不能作为全局资源根节点。

这套设计正好和资源模型对齐。

## phantom 类型参数

有些类型参数只用来表达“标记意义”，并不真正出现在字段里。Move 允许这类参数被声明成 phantom。一个典型例子是币种标识：

```move
module 0x2::m {
    struct CurrencyA {}
    struct CurrencyB {}

    struct Coin<phantom Currency> has store {
        value: u64,
    }
}
```

phantom 参数的好处是：你可以用类型来区分不同语义，但不会因为这些标记类型本身不具备某些能力，而拖累整体类型。

## 一个很实用的模式

泛型 + phantom 经常一起出现。你可以用泛型函数定义“通用逻辑”，再用 phantom 类型区分“同一逻辑下的不同实例”。这在代币、权限、状态标识和存储索引里都非常有用。

## 泛型先天是保守的

原书在这里强调了一个很重要的直觉：当类型参数还不知道具体是什么时，类型系统必须先按“最坏情况”处理。所以，泛型参数默认不带能力，除非你显式约束它。

```move
module 0x2::m {
    struct R {}

    fun unsafe_consume<T>(x: T) {
        // x 不能被自动丢弃，因为 T 没有 drop 约束
    }

    fun consume<T: drop>(x: T) {
        // 这里才允许把 x 当成可丢弃值
    }

    fun unsafe_double<T>(x: T) {
        (copy x, x)
    }

    fun double<T: copy>(x: T) {
        (copy x, x)
    }

    fun foo() {
        let r = R {};
        consume<R>(r);
    }
}
```

中文备注：原书用这些例子说明，泛型并不是“什么都能塞进去”，而是“你先告诉类型系统这个参数有什么能力”。没有约束时，编译器必须谨慎地拒绝那些可能不安全的操作。

## 类型参数约束是调用点检查的

```move
module 0x2::m {
    struct Foo<T: key> { x: T }

    struct Bar { x: Foo<u8> }
    struct Baz<T> { x: Foo<T> }
}
```

```move
module 0x2::m {
    struct S<T> has copy {
        f: T,
    }

    struct NoAbilities {}

    struct S2<T: copy + drop + store> {
        f: T,
    }
}
```

中文备注：原书强调，约束不是“声明时看起来合理就行”，而是会在调用点被严格验证。也就是说，你写出一个泛型函数或泛型结构体，只是建立了规则；真正实例化时，传入的类型必须真的满足这些规则。

## 能力的递归传播

能力不是只看顶层类型名，而是会沿字段递归传播。只要某个字段缺少必要能力，外层类型就不能“伪装”自己拥有它。

```move
module 0x42::example {
    struct NoAbilities {}

    struct Ignorable has drop { f: u64 }
    struct Pair has copy, drop, store { x: u64, y: u64 }

    struct WantsCopy has copy {
        f: NoAbilities,
    }

    struct MyResource has key {
        f: NoAbilities,
    }
}
```

中文备注：原书用这些反例说明，`copy`、`drop`、`store` 和 `key` 都不是装饰，而是对整个字段树的约束。如果底层字段不满足，外层就没有资格使用这个能力。

## 条件能力：泛型实例不一定拥有声明中的能力

这是原书最重要、也最容易被忽略的一段。一个泛型结构体声明了能力，不代表它的每个实例都拥有这些能力。

```move
module 0x42::example {
    struct S has copy, drop { f: bool }
    struct Cup<T> has copy, drop, store { item: T }

    fun example(c_x: Cup<u64>, c_s: Cup<S>) {
        let c_x2 = copy c_x;
        let c_s2 = copy c_s;
    }

    fun invalid(c_account: Cup<signer>, c_n: Cup<NoAbilities>) {
        let c_account2 = copy c_account;
        let c_n2 = copy c_n;
    }
}
```

```move
module 0x42::example {
    struct S has copy, drop { f: bool }
    struct Cup<T> has copy, drop, store { item: T }

    fun unused() {
        Cup<bool> { item: true };
        Cup<S> { item: S { f: false } };
    }

    fun invalid_unused() {
        Cup<NoAbilities> { item: NoAbilities {} };
    }
}
```

中文备注：原书在这里其实是在解释“条件能力”的核心逻辑：`Cup` 能不能 `copy`、`drop`、`store`，要看它装的是什么。`vector<T>` 之所以能和任何 `T` 一起工作，就是因为这些能力会随着 `T` 的能力一起变化。

## 递归限制和类型层递归

原书还专门提醒，泛型结构体不能递归地包含自己，类型层面的递归调用也不能无限生长。

```move
module 0x2::m {
    struct Foo<T> {
        x: Foo<u64>
    }

    struct Bar<T> {
        x: Bar<T>
    }
}
```

```move
module 0x2::m {
    struct A<T> {}

    fun foo<T>() {
        foo<A<T>>();
    }
}
```

中文备注：这部分是原书比较“硬”的类型系统边界。Move 之所以限制这种递归，不是为了少一点表达力，而是为了避免编译器、VM 和验证工具处理无穷类型时变得过于复杂。

## 泛型和能力的实践直觉

把这一章合起来看，你可以得到一个很稳定的判断方式：

- 先想这个类型参数会不会影响能不能复制、丢弃或存储。
- 再想你是否需要用约束把这些能力写进签名。
- 最后看这个泛型实例是否真的满足你允许的操作。

中文备注：原书的目标不是让你死记语法，而是让你形成一种“先想约束、再写代码”的习惯。对 Move 来说，这种习惯比单纯记忆 API 更重要。

## 原书更强调的三个泛型/能力细节

### 1. 结构体的能力会被字段递归决定

```move
module 0x2::m {
    struct Wrapper<T> {
        value: T,
    }
}
```

中文备注：原书想让你掌握的不是“写 `has copy` 就完了”，而是“能力是否成立要看字段是否也满足同样约束”。如果字段里装着一个不能复制或不能丢弃的值，那外层类型也不能假装自己更自由。

### 2. `Coin<Currency>` 这种写法是在表达语义标签

```move
module 0x2::m {
    struct Currency1 {}
    struct Currency2 {}

    struct Coin<phantom Currency> has store {
        value: u64,
    }

    public fun mint_generic<Currency>(value: u64): Coin<Currency> {
        Coin { value }
    }
}
```

中文备注：这里最重要的不是“泛型好像很灵活”，而是“类型本身可以携带语义标签”。原书拿货币举例，是因为它非常适合说明“同样是 coin，但属于不同币种的 coin 不能混用”。

### 3. 有些类型参数完全可以不用在字段里出现

```move
module 0x2::m {
    struct Foo<T> {
        foo: u64,
    }
}
```

中文备注：这说明泛型不只是“字段里必须出现这个类型”。有时候它只负责给类型加一个额外的语义维度。原书用这一点来说明，泛型参数本身就是一种设计工具，而不是纯粹为了减少重复。
