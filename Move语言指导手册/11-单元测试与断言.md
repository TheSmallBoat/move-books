# 11 单元测试与断言：如何验证行为

写 Move 代码时，只保证“能编译”是不够的。因为 Move 的很多错误不是语法错误，而是资源语义、断言条件和全局存储状态没有满足。单元测试和断言，就是你最常用的第一层验证工具。

## 测试注解

Move 提供三类常见测试注解：

- `#[test]`：把函数标记为测试。
- `#[test_only]`：只在测试构建中包含的代码。
- `#[expected_failure]`：声明该测试预期失败。

`#[test]` 在无参数形式下，只能用于没有参数的函数。带参数时，参数必须是 `signer` 类型，并且通常通过注解把地址传进来。

```move
module 0x42::example {
    #[test]
    fun smoke_test() {
        assert!(true, 0);
    }

    #[test(addr = @0xC0FFEE)]
    fun signer_test(addr: signer) {
        assert!(std::signer::address_of(&addr) == @0xC0FFEE, 0);
    }
}
```

## 预期失败

`#[expected_failure]` 很适合用来检查“应该失败”的行为，比如断言失败、算术错误、向量越界或明确的 abort。

```move
module 0x42::example {
    #[test]
    #[expected_failure(abort_code = 0)]
    fun zero_should_fail() {
        assert!(false, 0);
    }
}
```

你可以把它理解成：测试不仅能证明“成功时对”，也能证明“失败时确实会失败”。这对安全代码特别重要。

## `abort` 和 `assert!`

`abort` 会直接中止当前事务，并回滚这次事务里已经做过的全局状态修改。它可以接受一个 `u64` 错误码，或者在较新的版本里接受 `vector<u8>` 消息。

`assert!` 则是更常用的包装：条件成立时继续执行，条件不成立时中止事务。

```move
assert!(balance >= amount, 42);
assert!(balance >= amount);
assert!(balance >= amount, b"insufficient balance");
```

几个细节要特别记住：

- 不带错误码的 `assert!` 会使用默认的未指定错误码。
- 带消息的 `abort` / `assert!` 要求消息是合法 UTF-8，且不能太长。
- `assert!` 不是普通函数，而是编译器提供的宏式构造。

## `assert_eq!` 和 `assert_ne!`

如果你的工具链版本足够新，还可以使用 `assert_eq!` 和 `assert_ne!`。它们适合比较两个值是否相等或不等，并且在失败时能给出更清晰的信息。

这类宏在测试里非常有用，因为它们能把“为什么失败”说得更明确，而不是只给你一个条件为假的抽象报错。

## 一个小例子

```move
module 0x1::coin {
    struct MyCoin has key { value: u64 }

    public fun make_sure_non_zero_coin(coin: MyCoin): MyCoin {
        assert!(coin.value > 0, 0);
        coin
    }

    #[test]
    fun make_sure_non_zero_coin_passes() {
        let coin = MyCoin { value: 1 };
        let MyCoin { value: _ } = make_sure_non_zero_coin(coin);
    }

    #[test]
    #[expected_failure(abort_code = 0, location = Self)]
    fun make_sure_zero_coin_fails() {
        let coin = MyCoin { value: 0 };
        let MyCoin { value: _ } = make_sure_non_zero_coin(coin);
    }
}
```

这个例子把“成功路径”和“失败路径”都覆盖了。写 Move 测试时，这种双向验证很值得养成习惯。

## 测试注解再补完整一点

原书对测试注解的说明很细，核心是三类：`#[test]`、`#[test_only]` 和 `#[expected_failure]`。

```move
module 0x42::example {
    #[test]
    fun this_is_a_test() { }

    #[test]
    #[expected_failure]
    public fun this_test_will_abort_and_pass() { abort 1 }

    #[test]
    #[expected_failure(vector_error, minor_status = 1, location = Self)]
    fun borrow_out_of_range() { }
}
```

```move
#[test_only]
module 0x42::abc { /* ... */ }

module 0x42::other {
    #[test_only]
    address ADDR = @0x1;

    #[test_only]
    use 0x1::some_other_module;

    #[test_only]
    struct SomeStruct { }

    #[test_only]
    fun test_only_function() { }
}
```

中文备注：原书把 `#[test_only]` 用在模块、地址、`use`、结构体和函数上，就是为了说明测试支持代码可以单独存在，不会污染正式编译产物。这对复杂测试辅助函数尤其重要。

## 带参数的测试更像真实交易

```move
module 0x42::example {
    #[test(arg = @0xC0FFEE)]
    fun this_is_correct_now(arg: signer) { }

    #[test(a = @0xC0FFEE, b = @0xCAFE)]
    fun this_works(a: signer, b: signer) { }

    #[test(arg = @TEST_NAMED_ADDR)]
    fun this_is_correct_now_named(arg: signer) { }

    #[test_only]
    address TEST_NAMED_ADDR = @0x1;
}
```

中文备注：原书特别强调，测试参数只支持 `signer`，并且参数名必须一一对应。这让测试更像真实交易里的多方签名场景，也避免了把不合法的参数偷偷塞进测试函数。

## 运行测试时会发生什么

原书对 `aptos move test` 的输出做了很明确的说明：测试结果会显示 `PASS`、`FAIL` 或 `TIMEOUT`，而且默认是并行执行的，所以顺序不一定稳定。

```text
$ aptos move test
BUILDING MoveStdlib
BUILDING TestExample
Running Move unit tests
[ PASS    ] 0x1::my_module::make_sure_non_zero_coin_passes
[ PASS    ] 0x1::my_module::make_sure_zero_coin_fails
[ PASS    ] 0x1::my_module::test_has_coin
Test result: OK. Total tests: 3; passed: 3; failed: 0
```

中文备注：这段输出值得你在团队里熟悉，因为它能直接告诉你测试是成功、失败还是超时。对于 Move 来说，超时往往意味着循环、路径数量或测试输入组合太复杂了。

## 断言消息的细节再补一层

```move
script {
    fun demo(idx: u64, len: u64) {
        assert!(idx < len, b"index {} out of bounds for vector of length {}", idx, len);
        assert_eq!(idx, len, b"mismatch for key {}", idx);
        assert_ne!(idx, 0);
    }
}
```

中文备注：原书的意思不是“你必须把所有失败都写成消息”，而是提示你：当失败原因和上下文一起出现时，定位问题会快很多。尤其在资源和集合边界上，清楚的断言信息非常有价值。

## 原书里更完整的断言形式

### 1. 格式化断言消息

```move
script {
    fun demo(idx: u64, len: u64) {
        assert!(idx < len, b"index {} out of bounds for vector of length {}", idx, len);
    }
}
```

中文备注：原书强调过，格式化消息的意义不是“炫技”，而是让失败信息直接告诉你哪里错了。对调试来说，这比只给一个数字错误码更有帮助。

### 2. `assert_eq!` 和 `assert_ne!`

```move
script {
    fun demo(actual: u64, expected: u64, key: u64) {
        assert_eq!(actual, expected);
        assert_eq!(actual, expected, b"mismatch for key {}", key);
        assert_ne!(actual, 0);
    }
}
```

中文备注：原书把这两个宏和 Rust 的同名宏类比得很清楚。它们的优点是会把左右两边的值都带进失败信息里，所以你不需要再额外打印上下文。

### 3. 多个 signer 测试参数

```move
module 0x42::example {
    #[test(a = @0xC0FFEE, b = @0xCAFE)]
    fun test_two_signers(a: signer, b: signer) {
        assert!(std::signer::address_of(&a) != std::signer::address_of(&b), 0);
    }
}
```

中文备注：这个例子说明测试不只是“单人验证”。当你的逻辑涉及多方账户时，测试参数可以帮助你模拟多个签名者，从而覆盖更贴近真实交易的场景。

### 4. 预期失败可以指定更细的错误类型

原书还展示了更细粒度的 `expected_failure` 写法，例如按 `vector_error`、`minor_status` 或 `location` 区分失败来源。对需要精确验证错误位置的测试来说，这比只写“会失败”要强得多。
