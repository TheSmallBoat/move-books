# 12 规格与证明：给程序写可验证的说明

如果测试是在挑选样本，那么规格和证明是在系统地检查“所有可能情况”。Move 的规格语言（MSL）和 Move Prover，就负责做这件事。

## 先理解目标

MSL 不是运行时代码，而是用来描述程序行为的说明语言。Move Prover 会静态检查这些说明是否和程序一致。它的价值在于“穷尽性”：不是只看几个测试输入，而是检查所有可能输入和状态组合。

这也是为什么规格特别适合安全敏感代码：你关心的往往不是“这次跑通了没有”，而是“任何合法输入下都会不会出错”。

## 先从前后置条件开始

对新手来说，最有用的入口通常是：

- `requires`：调用前必须满足什么。
- `ensures`：调用后必须成立什么。
- `aborts_if`：什么情况下允许中止。
- `invariant`：某个状态范围内始终成立什么。

你不需要一开始就掌握所有证明技巧。先把函数行为说清楚，再逐步补全证明约束，通常就足够有用。

## MSL 的关键差异

MSL 和普通 Move 有几处很重要的不同：

- 整数会以更适合证明的方式解释，常见的是 `num` 和 `bv` 两套表示。
- 在规格里，引用和普通值通常会被看成等价的值语义。
- 规格表达式只支持 Move 的一部分语法，再加上一些专门的规格构造。

你可以把它理解成：MSL 不是“再写一遍 Move”，而是“用一套更适合证明的语言去描述 Move”。

## 常见构造

规格里经常会看到这些东西：

- `forall` 和 `exists`：量化表达式。
- `choose`：从满足条件的值里挑一个。
- `old(...)`：表示进入函数前的状态。
- `global<T>(addr)`、`exists<T>(addr)`：关于全局资源的规格函数。

这些构造的共同目的，是把“程序执行前后发生了什么”说得足够精确。

## 一个很短的例子

```move
spec module 0x42::counter {
    spec fun get(addr: address): u64 {
        ensures result >= 0;
    }
}
```

这类规格并不复杂，但它已经足以表达一些基本保证。对新手来说，写规格的第一步不是证明复杂定理，而是养成“把函数的输入、输出和副作用写清楚”的习惯。

## 什么时候值得引入证明

如果你的代码只是简单工具函数，测试可能就够了。但如果你的代码涉及：

- 资源发布和删除
- 余额、授权、转移规则
- 复杂状态机
- 多个模块之间的协议

那么规格和证明就很值得提前介入。它们能帮你发现“单个测试看不出来，但整体语义已经偏了”的问题。

## 原书里更完整的规格例子

### 1. `requires`、`ensures` 和 `aborts_if` 一起看

```move
module 0x42::bank {
    public fun withdraw(balance: u64, amount: u64): u64 {
        if (amount > balance) abort 1;
        balance - amount
    }

    spec withdraw {
        requires amount <= balance;
        ensures result == old(balance) - amount;
        aborts_if amount > balance;
    }
}
```

中文备注：原书想让你形成的直觉是：前置条件描述“什么情况下函数应该被调用”，后置条件描述“调用之后必须成立什么”，而 `aborts_if` 则专门描述“什么情况下允许失败”。这三者合起来，才比较接近一个完整的行为契约。

### 2. `old(...)` 用来描述调用前状态

```move
spec withdraw {
    ensures result <= old(balance);
}
```

中文备注：`old(...)` 的作用很像“快照”。原书在讲 Move Prover 时反复使用它，是因为很多安全性质都不是“现在是什么”，而是“调用前后如何变化”。

### 3. 量化表达式用来描述整体性质

```move
spec module 0x42::v {
    spec fun all_non_negative(v: vector<u64>): bool {
        forall i in 0..len(v): v[i] >= 0
    }
}
```

中文备注：量化表达式的价值在于，它让你不用手工枚举每个元素。原书用它来说明，Move 规格不是只能描述单次读写，还能描述“整个集合都满足某种性质”。

### 4. 规格语言和程序语言不是一回事

原书特别强调，MSL 的整数、引用和若干表达式语义和普通 Move 不完全相同。你写规格时，思维模型应当是“在证明语境里描述程序”，而不是“把程序再写一遍”。

## 行为谓词：高阶函数的契约工具

原书把行为谓词单独提出来，是因为它们解决了一个很现实的问题：当函数参数本身也是函数时，外层函数怎么描述内层函数的行为？

| 谓词 | 含义 |
| --- | --- |
| `ensures_of<f>(args, result)` | 函数 `f` 在这些参数和返回值上的后置条件 |
| `aborts_of<f>(args)` | 函数 `f` 在这些参数上的中止条件 |
| `requires_of<f>(args)` | 函数 `f` 的前置条件 |
| `result_of<f>(args)` | 满足后置条件的确定性结果 |

```move
fun apply(f: |u64| u64, x: u64): u64 {
    f(x)
}

spec apply {
    ensures ensures_of<f>(x, result);
}

fun apply_may_abort(f: |u64| u64, x: u64): u64 {
    f(x)
}

spec apply_may_abort {
    aborts_if aborts_of<f>(x);
    ensures ensures_of<f>(x, result);
}

fun apply_no_abort(f: |u64| u64, x: u64): u64 {
    f(x)
}

spec apply_no_abort {
    requires !aborts_of<f>(x);
    aborts_if false;
    ensures ensures_of<f>(x, result);
}
```

中文备注：原书想让你意识到，契约可以被“带着走”。外层函数不需要知道闭包的实现细节，只要引用行为谓词，就能把闭包的前后条件传播出去。

### `result_of` 和顺序调用

```move
fun apply_seq(f: |u64| u64 has copy, x: u64): u64 {
    f(f(x))
}

spec apply_seq {
    let y = result_of<f>(x);
    requires requires_of<f>(x) && requires_of<f>(y);
    aborts_if aborts_of<f>(x) || aborts_of<f>(y);
    ensures result == result_of<f>(y);
}
```

中文备注：`result_of` 的价值在于，它让你可以把“第一次调用的返回值”当作中间结果写进规格里。这在重复应用、循环不变量和 fold 类推理中尤其有用。

### 透明和 opaque 的差别

```move
fun apply_opaque(f: |u64| u64, x: u64): u64 {
    f(x)
}

spec apply_opaque {
    pragma opaque = true;
    ensures ensures_of<f>(x, result);
}

fun test_opaque(x: u64): u64 {
    apply_opaque(|y| y + 5 spec { ensures result == y + 5; }, x)
}

spec test_opaque {
    ensures result == x + 5;
}
```

中文备注：`pragma opaque` 的意思是“只看规格，不看实现”。原书把这个概念和模块化验证绑在一起，是因为当高阶函数的实现不可见时，外部只能依赖契约工作。

### `&mut` 参数的行为谓词

```move
fun apply_mut(f: |&mut u64| u64, x: &mut u64): u64 { f(x) }

spec apply_mut {
    ensures result == result_of<f>(x);
}

fun apply_void_mut(f: |&mut u64|, x: &mut u64) { f(x) }

spec apply_void_mut {
    ensures ensures_of<f>(x);
}

fun apply_two_results(f: |&mut u64| (u64, u64), x: &mut u64): (u64, u64) {
    f(x)
}

spec apply_two_results {
    ensures (result_1, result_2) == result_of<f>(x);
}
```

中文备注：原书专门强调 `&mut` 只出现一次，证明器会自动处理它的前后状态。这样你在规格里写的不是“两个独立的值”，而是同一个状态的前后快照。

## 访问说明和 frame conditions

当高阶函数可能读写全局状态时，证明器还要知道它可以触碰哪些资源，这就需要 `reads_of` 和 `modifies_of`。

```move
spec apply {
    pragma opaque;
    reads_of<f> Config;
    modifies_of<f>(a: address) Data[a];
    ensures ensures_of<f>(x, result);
    aborts_if aborts_of<f>(x);
}
```

中文备注：原书把这一段称作 frame condition 的一部分，因为它决定了“哪些状态会保持不变”。如果不把这些边界说清楚，证明器就无法判断调用前后到底哪些资源应该被视为稳定的。

## 证明块：拆成可验证的小步

```move
fun double_plus_one(x: u64): u64 {
    2 * x + 1
}

spec double_plus_one {
    requires 2 * x + 1 <= MAX_U64;
    ensures result >= x;
} proof {
    calc(
        2 * x + 1
        >= 2 * x
        >= x
    );
}
```

```move
fun abs_diff(a: u64, b: u64): u64 {
    if (a >= b) { a - b } else { b - a }
}

spec abs_diff {
    ensures result == if (a >= b) { a - b } else { b - a };
} proof {
    split a >= b;
}
```

```move
fun square_plus_one(x: u64): u64 {
    (x + 1) * (x + 1)
}

spec square_plus_one {
    requires x + 1 <= 4294967295;
    ensures result == (x + 1) * (x + 1);
} proof {
    let y = x + 1;
    let r = y * y;
    assert r == (x + 1) * (x + 1);
    post assert r == result;
}
```

中文备注：原书把 `calc`、`split`、`assert` 和 `post` 分开讲，是因为它们分别擅长不同类型的推理：代数重写、分支分析、局部事实和返回点推理。理解这一点后，你会更容易选择“哪一种证明工具最贴合当前问题”。

### `assume [trusted]`、`lemma` 和 `apply`

```move
fun div3_le(x: u64): u64 { x / 3 }

spec div3_le {
    ensures result <= x;
} proof {
    assume [trusted] x / 3 <= x;
}
```

```move
spec lemma monotonicity(x: num, y: num) {
    requires 0 <= x;
    requires x <= y;
    ensures sum(x) <= sum(y);
} proof {
    if (x < y) {
        assert sum(y - 1) <= sum(y);
        apply monotonicity(x, y - 1);
    }
}
```

```move
fun sum_up_to(n: u64): u64 {
    if (n == 0) { 0 }
    else { n + sum_up_to(n - 1) }
}

spec sum_up_to {
    aborts_if sum(n) > MAX_U64;
    ensures result == sum(n);
} proof {
    forall x: num, y: num {sum(x), sum(y)} apply monotonicity(x, y);
}
```

中文备注：原书非常明确地提醒，`assume [trusted]` 是一种“信任证明者”的做法，不能滥用；`lemma` 则是把常见推理抽象成可复用定理；`apply` 和 `forall...apply` 让这些定理在具体或量化场景下被实际使用。

## 带 `&mut` 的证明和 `old()`

```move
struct Counter has drop {
    value: u64,
}

fun increment(c: &mut Counter) {
    c.value = c.value + 1;
}

spec increment {
    requires c.value < MAX_U64;
    ensures c.value == old(c.value) + 1;
} proof {
    assert c.value < MAX_U64;
    assert c.value + 1 <= MAX_U64;
}

fun add_and_return(c: &mut Counter, n: u64): u64 {
    c.value = c.value + n;
    c.value
}

spec add_and_return {
    requires c.value + n <= MAX_U64;
    ensures c.value == old(c.value) + n;
    ensures result == c.value;
} proof {
    assert c.value + n <= MAX_U64;
    post assert c.value == old(c.value) + n;
    post assert result == c.value;
}
```

中文备注：这部分特别适合新手建立“证明也会看状态前后变化”的直觉。对 Move 来说，资源、可变引用和返回值都不是抽象符号，而是可以在规格里明确追踪的状态对象。

## 弱前置推导：让证明器帮你补规格

原书最后还讲了规格推导引擎：它会沿着字节码做 weakest-precondition 分析，从函数出口往回推，推导出需要的 `ensures`、`aborts_if` 和 frame 条件。

中文备注：你可以把它理解成“从结果倒推前提”的自动化工具。它特别适合直线代码和资源变更明显的函数，但遇到循环时，仍然需要你先写出循环不变量。也就是说，自动推导很强，但它仍然需要人类给出关键约束。
