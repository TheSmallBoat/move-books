# 13 编码规范与实践：写给未来读者的代码

Move 的语法并不复杂，但写得清楚和写得“只是能跑”之间，差别很大。这一章的目标不是规定唯一正确的风格，而是给出一套更适合阅读、审计和维护的实践习惯。

## 命名

遵循最基本的命名约定，几乎总是值得的：

- 模块名：小写，通常用 `snake_case`。
- 类型名：大写驼峰。
- 函数名：小写 `snake_case`。
- 常量名：通常用全大写，错误码常量以 `E` 开头。

如果你的命名规则让人一眼就看不出“这是什么类型、这是什么函数、这是什么常量”，那通常就该重命名。

## 导入

一个很稳妥的习惯是：

- `use` 放在模块顶部。
- 类型优先导入，函数则尽量用模块路径全限定调用。
- 如果发生命名冲突，再用 `as` 重命名。

这样做的好处是，读者能清楚分辨“来自哪个模块的类型”与“来自哪个模块的行为”。

## 注释

文档注释 `///` 应该紧贴它所说明的条目。对模块、结构体和公开函数，写清楚“它做什么、为什么存在、有哪些前提”都很重要。

一个经验法则是：

- 公共 API 写目的和不变量。
- 复杂逻辑写关键假设。
- 资源操作写清楚副作用。

如果你的注释只是重复代码本身，那通常没什么价值。

## 格式

Move 社区常见的格式习惯包括：

- 使用四空格缩进。
- 行不要过长。
- 结构体和常量通常放在函数前面。
- 顶层声明尽量按“类型、常量、函数”的顺序组织。

这些规则不是为了机械统一，而是为了让代码块更容易扫描。

## 实践建议

真正写 Move 时，最值得坚持的几条实践是：

- 把资源操作封装在定义资源的模块里。
- 对外暴露尽量小的公共接口。
- 让 `acquires`、`assert!` 和错误码表达清楚你的意图。
- 不要为了少写几行代码而牺牲资源语义的清晰度。

如果一段代码的“为什么”很难讲清楚，那往往说明它的 API 还不够好。

## 原书里特别强调的导入与注释规则

### 1. `use` 放顶部，类型和函数分开处理

```move
    module 0x1::bar {
        use 0x1::foo::{Self, Foo};

        public fun do_bar(x: u64): Foo {
            if (x == 10) {
                foo::do_foo()
            } else {
                abort 0
            }
        }
    }
    ```

    ```move
    module 0x1::importer {
        use 0x1::other_foo::Foo as OtherFoo;
        use 0x1::foo::Foo;
    }
```

中文备注：原书的意思很直接：函数最好保持模块限定调用，类型则尽量在顶部导入。这样一来，代码读者更容易分辨“依赖了什么类型”和“调用了什么行为”。

### 2. 每个模块、结构体和公开函数都应该写注释

```move
    /// My awesome module, doc comment can be used here
    module 0x42::example {
        // double slash can be anywhere

        /// My awesome constant
        const MY_VALUE: u64 = 5;

        /// My awesome error message
        const E_MY_ERROR: u64 = 10;

        #[view]
        /// My awesome view function
        fun show_me_the_money() {
            // ...
        }

        /* Similarly block comments can be anywhere */
    }
```

```move
    module 0x42::example {
        /// My awesome view function <- must be below the annotation, right above the thing commented
        #[view]
        fun show_me_the_money() {
            // ...
            /// Within a function
        }

        /// Not attached to anything
    }
```

中文备注：原书在这里不是在讲“注释很多才高级”，而是在讲“注释位置要正确”。文档注释必须紧贴被说明的项，否则工具和读者都会失去它的指向性。

### 3. UTF-8 中文注释是可以的

```move
    /// 这是一个中文注释示例
    module 0x42::example {
        /// 余额资源
        struct Balance has key { value: u64 }
    }
```

中文备注：原书说明，从较新的 Aptos CLI 开始，UTF-8 字符可以出现在注释里。这对中文手册尤其有价值，因为它允许你直接写清楚语义，不必为了兼容性而绕回英文。

## 命名再加一点原书风格

```move
    module my_module {}
    module foo_bar_42 {}

    script {
        const FLAG: bool = false;
        const MY_ERROR_CODE: u64 = 0;
        const ADDRESS_42: address = @0x42;
    }
```

中文备注：原书的建议并不是要你死守某一种命名风格，而是希望你让名字本身就传达用途。模块、类型、函数和常量的命名只要稳定一致，读者就能更快建立心智模型。

## 原书里很强调的三个实践习惯

### 1. 错误码命名要稳定

```move
    module 0x42::bank {
        const E_INSUFFICIENT_BALANCE: u64 = 0;
        const E_NOT_FOUND: u64 = 1;
    }
```

中文备注：原书的风格建议很明确：错误码通常用大写前缀，便于审计和搜索。你在看 Move 项目时，只要看到这种命名，就能很快知道这是“错误码”，不是普通业务常量。

### 2. 资源操作尽量封装在定义模块里

```move
    module 0x42::bank {
        /// 存储在账户地址中的余额资源。
        struct Balance has key { value: u64 }

        /// 只有这个模块知道如何创建、读取和删除 Balance。
        public fun deposit(account: &signer, value: u64) acquires Balance {
            move_to(account, Balance { value })
        }
    }
```

中文备注：原书反复强调这一点，不只是为了代码整洁，而是为了保护不变量。把资源的 pack/unpack 权限收在定义模块里，外部就更难绕过你的规则。

### 3. 文档注释应该直接说明副作用

```move
    /// 从账户中读取余额，不修改任何全局状态。
    public fun get_balance(addr: address): u64 acquires Balance {
        borrow_global<Balance>(addr).value
    }
```

中文备注：原书建议给模块、结构体和公开函数都写注释，原因很简单：Move 代码的关键不在“这行做了什么”，而在“这段代码允许哪些状态变化”。如果注释能直接写出副作用，读者会省很多时间。
